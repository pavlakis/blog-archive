Moving a Website to a new Provider

Details
Created on Sunday, 27 March 2011 22:35

Moving a website can be a tricky task and it usually helps being extra careful than having an unsatisfied customer. Below, I will just outline a few simple steps I take to make sure that everything goes smoothly. First of all, when you decide to move server or hosting provider, give yourself a bit of extra time and get the new package first. This way, you will have enough time to do the switch with less stress; especially if you don’t know the environment or control panel being used.

I normally use a different provider for my DNS settings which is usually the same place I purchase my domains from. This way, when I need to change hosting provider, I don’t need to stress about the move. Just setup in the new place, and then do the switch.

So, let’s get started. I take it that most people these days use database driven websites. This adds an extra layer of complexity. That’s ok, nothing to worry about we will just go over the implications and make sure we cover them also. Let’s say you have a Content Management System or blog (eg Drupal, Joomla, Wordpress, etc). As long as you have users using the website, eg adding articles, comments or other content, you need to make sure that when you make the transfer, the information you are passing is up-to-date.

My suggestion is, disable all users – apart from the admin – and export the database. This way, users can still view the content and you are certain to have the latest info when making the move. On the new provider, create first the database. It is usually best practice to create a new database user with only the necessary access to the database. In general you will need: Select, Update, Insert Delete, but some systems may need more – you will need to check for that yourself.

At this point you can add to your configuration file ( if it is Joomla, that’s in configuration.php) the new configuration eg DB Host, DB Name, Username, Password. If you are using UTF-8, open the database sql file that you’ve exported, and add the following line on the top:
SET NAMES 'utf8';
Also, if you are using the command line to import the sql, it is usually safer to let the DB which database to use (this should go on the top of everything):
use my_db_name;

Next, we can sort out the code base. This step is straightforward although you should always be careful with the file permissions. Some providers have high security which you will need to change yourself. I will not be covering permissions here.

Now, this is the most important step. Test your new configuration. On your new provider, whether you’ve got access to the VirtualHost configuration, or it is done automatically by a control panel, you will have an IP address pointing to your domain. At this point you are not pointing to that one. Your website is still live with your old provider.

To test it on the new one, you will need to add an entry to your hosts file. That will be something like: 192.168.4.1 www.mysite.com Now you should be able to load and test your website on your new provider.

In most cases you don’t need to do much, but I have had to make changes to .htaccess especially when using mod_rewrite. Once you are happy that everything works, re-activate all user accounts and change your DNS entries to point to your new domain.
